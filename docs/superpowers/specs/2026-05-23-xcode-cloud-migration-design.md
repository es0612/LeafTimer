# Xcode Cloud TestFlight 移行 (Issue #13 Part B-E) Design

> Generated by `superpowers:brainstorming` skill on 2026-05-23. Source: Issue #13. Implementation plan to follow via `superpowers:writing-plans` on the same branch (`feature/issue-13-part-b-e-xcode-cloud`).

## 目的

LeafTimer の TestFlight 配信を **fastlane beta lane** から **Xcode Cloud Workflow** に一本化する。Issue #13 の「やること #2-#6」(Issue #13 Part A は別 PR #24 で完了済み)。

- 開発者がローカルで `bundle exec fastlane beta` を打たなくても、git push (master or tag) で TestFlight に Build が上がる状態を作る
- 機密ファイル (`Keys.plist` / `GoogleService-Info.plist`、いずれも `.gitignore` 対象) を Xcode Cloud の Build 環境に安全に届ける仕組みを設計
- fastlane は緊急時 rollback 用に残し、`beta` lane だけ deprecated 化

## 主要決定

| 領域 | 採用案 | 却下案 (備考) |
|---|---|---|
| Secrets 受け渡し | **A: base64 Environment Variable** | B: private リポ fetch (個人開発で過剰) / C: Xcode Cloud File Secrets (調査コストあり、A で十分) |
| fastlane | **D-1: `beta` lane に deprecated コメント、`unittests` lane は無修正で残す** | 全削除 (rollback の保険を失う) / 完全並行運用 (保守 2 倍) |
| Build Number 管理 | **F-1: Xcode Cloud auto-increment (`CURRENT_PROJECT_VERSION`)** | GitHub Actions 事前 bump (CI 二重化) / 手動 (人為ミス) |
| MARKETING_VERSION | **手動** (リリースごと開発者編集、本 design スコープ外) | 自動 (リリース判断は人間が下すべき) |
| 移行戦略 | **III: 1 Plan / 1 PR で複数 commit** | I: Big Bang (PR レビュー負荷高) / II: 2 PR (段階的 merge、現状規模では過剰) |

## Architecture

### 移行前 (現状)

```
開発者ローカル
  └ bundle exec fastlane beta
      ├ pod install (前提)
      ├ scan (unittests)
      ├ increment_build_number
      ├ commit_version_bump → push_to_git_remote (master)
      ├ build_app (LeafTimer Release scheme)
      └ upload_to_testflight
```

### 移行後

```
git push (master / tag)
  └ Xcode Cloud Workflow trigger
      ├ ci_scripts/ci_post_clone.sh
      │   ├ KEYS_PLIST_BASE64        → base64 -d → app/Keys.plist
      │   ├ GOOGLE_SERVICE_INFO_PLIST_BASE64 → base64 -d → app/GoogleService-Info.plist
      │   └ cd app && pod install
      ├ Xcode が archive (LeafTimer Release scheme)
      ├ Build Number 自動 increment (Apple 側で管理)
      └ Post-Action: TestFlight 内部テスター配布
```

|---|---|---|
| `app/ci_scripts/ci_post_clone.sh` (新規) | Bash script | Xcode Cloud build 開始時に走る hook (`ci_post_clone.sh` は Apple Xcode Cloud の予約ファイル名)。Env Var から plist 復元 + `pod install` |
| `docs/ver1_2/xcode-cloud-setup.md` (新規) | docs | App Store Connect 側の設定手順 + Env Var encode コマンド + 動作確認チェックリスト + rollback 手順 |

| Xcode Cloud Workflow (Apple Developer 側) | UI 設定 | docs に手順記載のみ、コード変更なし。Start Condition / Build Action (Archive) / Post-Action (TestFlight) / Env Var 登録 |

| (既存・無変更) `Podfile` / `Podfile.lock` | 依存管理 | Xcode Cloud で `pod install` 実行時の再現性は `Podfile.lock` に依存、現状で OK |

## Data Flow (Secrets 受け渡し)

### 1. 開発者ローカル (1 回切り、Env Var 登録時のみ)

```bash
# Keys.plist を base64 化
base64 -i app/Keys.plist | pbcopy
# → App Store Connect の Workflow Environment Variables に
#    KEYS_PLIST_BASE64 として Secret 区分で登録

base64 -i app/GoogleService-Info.plist | pbcopy
# → GOOGLE_SERVICE_INFO_PLIST_BASE64 として同様に登録
```

`.plist` が更新されたら同手順で Env Var を上書き登録。

### 2. Xcode Cloud build 時 (`app/ci_scripts/ci_post_clone.sh` 内)

```bash
#!/usr/bin/env bash
set -euo pipefail

# Env Var → ファイル復元
echo "${KEYS_PLIST_BASE64}" | base64 -d > "${CI_WORKSPACE}/app/Keys.plist"
echo "${GOOGLE_SERVICE_INFO_PLIST_BASE64}" | base64 -d > "${CI_WORKSPACE}/app/GoogleService-Info.plist"

# CocoaPods 取得
cd "${CI_WORKSPACE}/app"
pod install
```

`$CI_WORKSPACE` は Xcode Cloud 標準環境変数 (リポジトリ clone 先のルート)。

## Error Handling

`ci_post_clone.sh` は `set -euo pipefail` で開始 (LeafTimer の pipefail rule とも整合)。

| 失敗ケース | 検出 | 対応 |
|---|---|---|
| `KEYS_PLIST_BASE64` 未設定 | `set -u` で参照時に即 fail | エラー stderr 出力 + exit (Xcode Cloud ログから一発で原因特定可) |
| `base64 -d` 失敗 (壊れた encode) | パイプ失敗 + `set -o pipefail` | exit、Apple のログで内容確認 |
| `pod install` 失敗 | exit code != 0 + `set -e` | exit、`Podfile.lock` の整合性疑い |
| 復元後の plist が読めない (内容不正) | Xcode build 段階で fail | Apple のログで Xcode 出力確認、`ci_script` に出力検証ロジック追加は将来対応 (YAGNI) |

### Rollback 方針

`fastlane beta` lane が deprecated コメント付きで残るため、緊急時は:

```bash
cd app && bundle exec fastlane beta
```

で従来通り TestFlight 送信可能。Xcode Cloud で致命的な問題が起きた時の保険。

## Testing (動作確認チェックリスト)

`docs/ver1_2/xcode-cloud-setup.md` に同チェックリストを記載:

- [ ] Workflow を App Store Connect で "Start build manually" で初回トリガー
- [ ] `ci_post_clone.sh` の各 step が Apple ログで pass (echo した step 名で確認)
- [ ] Archive 成功 (Workflow page で緑色 ✓)
- [ ] TestFlight に新 Build が表示される (App Store Connect → My Apps → LeafTimer → TestFlight)
- [ ] Build を internal tester に配布 (自分の Apple ID)
- [ ] インストール → 起動 → **本番広告 ID で広告表示** (Issue #13 Part A = PR #24 と組み合わせ検証。Debug は test ID、Release は本番 ID)
- [ ] クラッシュ報告なし (24 時間監視、App Store Connect Crashes)

`make tests` (ローカル) は本 design では影響なし、引き続き同じ動作。

## Scope 注意 (コード変更外の作業)

| 項目 | 場所 | 実施者 | タイミング |
|---|---|---|---|
| GitHub Repository ↔ Apple Developer 接続 | App Store Connect UI | ユーザー (Apple ID 認証) | 1 回切り、初回のみ |
| Xcode Cloud Workflow 作成 (Start Condition / Build Action / Post-Action) | App Store Connect UI | ユーザー (docs 通り) | 1 回切り、初回のみ |
| Env Var 登録 (`KEYS_PLIST_BASE64` / `GOOGLE_SERVICE_INFO_PLIST_BASE64`) | App Store Connect UI Secret | ユーザー | 初回 + plist 更新時 |
| 月次ビルド時間枠の確認 | Apple Developer Program | ユーザー | LeafTimer 規模 (年数リリース) なら 25h/月 で十分余裕 |

これらは PR コードに含まれない。docs に手順を残すことで、将来別開発者がレビュー / 引き継ぎ可能にする。
